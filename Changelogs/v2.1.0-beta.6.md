# v2.1.0-beta.6

Sixth public preview of the **2.1** line. The headline is internal: the
runtime engine now builds against **MachOSwiftSection 0.12.0-beta.5**,
whose Swift-interface extraction was split into layered modules
(`SwiftDeclaration` / `SwiftIndexing` / `SwiftPrinting` /
`SwiftSpecialization` / `SwiftInterface`). Around that, this build lands
a batch of fixes for proxy-reached engines (Mac Catalyst helper,
attached apps), centralizes the cross-process communication layer, and
preserves nested specialization identity in the sidebar. The in-house
coordinator and UI packages are now consumed from their published
remotes, so a clean clone resolves without the sibling checkouts.

This is a `beta` build — only clients that opted in via
**Settings → Updates → Include pre-release versions** receive it.

---

## Bug Fixes

### Proxy-reached client engines no longer run in the wrong process

The `RuntimeEngineRequest` unification regressed any engine reached
through `RuntimeEngineProxyServer` when the proxied engine is itself a
client of another process (the Mac Catalyst helper, attached apps).
Shared request handlers called `request.perform(on:)` directly, so the
local implementation ran inside the proxy *host* process —
`loadImage` `dlopen`'d `/System/iOSSupport/…` images on the wrong side
and surfaced as *"An unknown error occured"* on the remote peer.
Handlers now route through `engine.dispatch(_:)`: client engines forward
upstream while server / local engines keep executing locally.

Loading progress also never crossed the proxy hop, leaving remote hosts
stuck on *"Preparing…"* until completion. The hand-rolled
`objectsLoadingProgress` side channel is replaced by a
`RuntimeEngineProgressRequest` pipeline — requests ride a
`{progressToken, request}` envelope, progress flows back as token-routed
`progressEvent` pushes, and chained proxies relay automatically with no
per-command code. Token routing also stops concurrent loads from
cross-talking on the previously shared subject.

> Wire-format note: `runtimeObjectsInImage` now ships the progress
> envelope and the progress channel was renamed, so peers (Server /
> helper / injected dylib) must be rebuilt from this revision.

### Communication layer centralized and hardened

The per-transport receive loop is consolidated into a single
`RuntimeMessageChannel` so the four transports (LocalSocket, DirectTCP,
Bonjour, XPC) stop diverging, closing several failure modes — each
reproduced by a test first in the new `ConnectionTransportRegressionTests`:

- **No deadlock / head-of-line blocking.** Responses route inline;
  fire-and-forget handlers run on a serial tail and response-producing
  handlers run concurrently. A handler awaiting a reply over the same
  connection no longer deadlocks the loop, and a slow handler no longer
  blocks other in-flight requests.
- **Fail fast on unrouted requests.** A request that hits no handler, or
  whose handler throws, now replies with a nonce-routed error envelope
  (new `isError` flag) carrying the real message, instead of hanging on
  a nil timeout or surfacing an opaque `DecodingError`.
- **Trailing-message delivery.** An `NWConnection` chunk is consumed
  before acting on `isComplete`, and buffered frames drain before the
  dispatch stream finishes, so a final message coalesced with the FIN is
  still delivered.
- **Transport robustness.** DirectTCP gets a `.waiting` tolerance window
  (parity with Bonjour), `EINTR` is retryable in the socket recv loop,
  the local socket server's pending-handler state is locked, and the
  receive-buffer scan is now O(n) via a persisted offset.

Core-side, `RuntimeEngine` connection-state events are serialized
through a single consumer to stop reorderings, and the proxy server
drops stale push relays on reconnect to avoid duplicate pushes.

### Per-image ObjC listing drops imported protocols

`__objc_protolist` carries a `protocol_t` for every protocol whose
`@protocol` was in scope at compile time, so each framework redundantly
listed its dependencies' protocols (`NSObject`, `CALayerDelegate`, …).
With no authoritative defining-image signal, ownership is now attributed
via a one-time whole-shared-cache scan: an image owns a protocol unless a
carrier of it sits in the image's transitive dependency closure. Per-image
protocol lists are correspondingly tighter.

### Nested sidebar specialization identity preserved

Specialization identity moved off `RuntimeObject` and onto the sidebar
cell view-model, and derived nested specializations now register under a
dedicated case. Together these preserve the runtime specialization tree's
identity across sidebar rebuilds, so nested specializations no longer
collapse or re-shuffle when the list is rediffed. A related equality bug
is also fixed: `Candidate` equality now includes `id` and `isGeneric`
(via `@Equatable`), so two candidates differing only in those fields no
longer compare equal while hashing differently — fixing the
`RuntimeSpecializationCandidateTests` divergence.

### Text scene controller rebuilt on re-entry

The cached `ContentTextViewController` avoids the UXKit push-transition
flash when rebinding on every sidebar selection, but reusing it after the
content area had left the text scene rebound a controller whose view was
already displaced from the hierarchy. A fresh instance is now created when
entering from another scene; in-scene selection changes keep the cached
one.

### Export module metadata resolution

Fixes module metadata resolution during export (#72).

## Improvements

### Popovers presented through the UXPopover transition

The generation-options, MCP-status, background-indexing, and
specialization type-picker popovers now present through `.uxPopover` /
`.closeUXPopover` instead of the native `.asPopover` presentation, which
ignored `preferredContentSize` and mis-sized UXKit content (defaulting to
500×500). The generation-options row also raises its horizontal hugging
priority so its width stops being ambiguous against the outer stack.

## Build & Tooling

These are developer-facing only and do not affect shipped Release builds
(which keep the fixed `com.mxiris.runtimeviewer.service` identity):

- **Debug-arm64e gets a fully distinct identity** so it no longer collides
  with a plain Debug build — service `dev.arm64e.mxiris.runtimeviewer.service`,
  app `RuntimeViewer-Debug-arm64e` (`dev.JH.RuntimeViewer.arm64e`). The two
  launch-service targets are merged into one whose product name is selected
  per configuration, with the LaunchDaemon plist generated at build time
  from a single template. The mach-service name is resolved at compile time
  via `#if RUNTIMEVIEWER_ARM64E`, which reaches the injected
  `RuntimeViewerServer` dylib that has no readable Info.plist.
- DerivedData / SourcePackages are routed to a dedicated cache volume when
  present (falls back to a project-relative path on CI), keeping checkouts
  out of the project tree.
- The development team ID is set directly in the project instead of going
  through an external xcconfig / environment override.
